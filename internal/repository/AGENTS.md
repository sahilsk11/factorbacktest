# `internal/repository` — agent guidelines

Rules for any code that lands in this directory.

## Always use the JET-generated model + go-jet builder

When you add a new repository method that touches a Postgres table, the row type and the SQL builder MUST come from the auto-generated `internal/db/models/postgres/<schema>/{model,table}` files, not from a hand-written struct or hand-rolled SQL.

If the generated files don't exist for the table you're working with (typically because you just added a migration in this PR):

1. Apply the migration locally: `python tools/migrations.py up postgres`
2. Regenerate the JET models: `make db-models`
3. Commit the regenerated files alongside your repository code

CI runs `make db-models-check`, which will fail if the committed files don't match what JET produces against the live schema. So skipping this step doesn't just mean stylistic inconsistency — it means CI will reject your PR.

### Why

- One source of truth for column names and types. Renaming a column in a migration causes a single Go compile error in the repository (because the column field on the JET table value vanished), not several scattered string-typed bugs that surface only at query time.
- Hand-rolled SQL with `$1, $2, ...` doesn't get checked against the schema at compile time. The go-jet DSL does — `t.Email.EQ(postgres.String(email))` won't compile if `Email` got renamed to `EmailAddress`.
- The generated row type already has the right `*time.Time` / `*string` nullability and the right `int32` widths for `INTEGER` columns. Re-stating those in a separate hand-rolled struct introduces drift.

## Patterns

The canonical example for the auth-style "small, hand-managed table with a few CRUD methods" shape is [auth_session.repository.go](auth_session.repository.go). Mirror it: import `model` + `table`, use `t.SELECT(t.AllColumns).WHERE(...)` etc.

For things go-jet's DSL doesn't express directly (e.g. `column = func(column)` updates), prefer the function helpers in `github.com/go-jet/jet/v2/postgres` (e.g. `GREATEST`, `LEAST`, `COALESCE`, `IntegerExpression.SUB(...)`) before falling back to `postgres.RawStatement`. Raw SQL is a last resort and should be commented with the reason it was needed.

## Mocks

Every repository interface should have a mock generated by the line in `Makefile`'s `mocks:` target. After adding a new repository file, add a corresponding `mockgen -source=... -destination=...` line and run `make mocks` to regenerate. Tests in other packages depend on the mocks; PRs that add a new repository without a mock will break test compilation in those packages.

## Errors

Sentinel errors for "row not found" should be exported and wrapped with `errors.Is`-friendly comparisons in callers. Don't return `sql.ErrNoRows` or `qrm.ErrNoRows` upward — translate at the repository boundary. See `ErrSessionNotFound` in [auth_session.repository.go](auth_session.repository.go) and `ErrEmailOTPNotFound` in [email_otp.repository.go](email_otp.repository.go).
